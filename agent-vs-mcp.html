AI Engineering — Fondamentaux

Agent · MCP · API

Trois concepts liés mais distincts — comprendre où chacun commence et où il s'arrête.

01 — Agent vs MCP en une phrase

Agent

🤖 Un système autonome qui raisonne et agit

Il reçoit un objectif, planifie, appelle des outils, observe les résultats et itère.

  • LLM comme moteur de décision
  • Mémoire (contexte, historique)
  • Choisit quoi faire et quand
  • Boucle : plan → action → observation

MCP — Model Context Protocol

🔌 Un protocole standard de communication

Il définit le "langage" que tout agent utilise pour parler à des outils — sans adaptateur sur mesure.

  • Standard d'interface (comme USB-C)
  • Pas d'intelligence propre
  • Expose : outils, ressources, prompts
  • Découplage total : agent ↔ outils

Analogie

L'agent = le technicien qui décide quels outils sortir et dans quel ordre.
MCP = la norme USB-C : une prise universelle — les outils s'y branchent sans câble propriétaire.

02 — MCP vs API : la différence

Réponse courte : MCP utilise les APIs — c'est une couche au-dessus.

Une API existe déjà (Slack, GitHub, Postgres…). MCP est le standard qui permet à l'agent de s'y brancher sans coder une intégration sur mesure à chaque fois.

Sans MCP

Chaque outil = intégration custom

  • Agent → code custom → API Slack
  • Agent → code custom → API GitHub
  • Agent → code custom → API PostgreSQL
  • 10 outils = 10 intégrations différentes

Avec MCP

Toujours la même interface

  • Agent → MCP → Slack MCP Server → API Slack
  • Agent → MCP → GitHub MCP Server → API GitHub
  • Agent → MCP → Postgres MCP Server → API PG
  • 10 outils = toujours le même protocole

Analogie

API = chaque fabricant a sa propre prise propriétaire.
MCP = USB-C. Les fabricants adaptent leur prise en interne, toi tu branches toujours le même câble.

03 — Les 3 couches de MCP

1 — La doc
→ dans le prompt du LLM

Le serveur annonce ce qu'il sait faire

Au démarrage, le serveur MCP envoie la liste de ses outils à l'agent. C'est un JSON qui atterrit dans le contexte du LLM — il le "lit" et comprend ce qu'il peut appeler.

// Ce que le LLM reçoit dans son contexte : { name: "search_web", description: "Searches the web for a query", parameters: { query: string } }
2 — Le protocole
→ tourne dans l'agent ET le serveur

Le format fixe des messages entre les deux

Ce n'est pas juste un fichier lu — c'est du code actif (un SDK) installé des deux côtés. Il garantit que les messages ont toujours la même forme, peu importe l'agent ou l'outil.

// L'agent appelle toujours comme ça : { method: "tools/call", params: { name: "search_web", arguments: { query: "MCP" } } } // Le serveur répond toujours comme ça : { content: [{ type: "text", text: "résultat..." }] }
3 — Le serveur
→ process qui tourne en local ou réseau

Le vrai code métier qui s'exécute

C'est un vrai programme (Node.js, Python…) qui reçoit les appels du protocole et exécute la vraie logique : lire un fichier, requêter une BDD, appeler une API.

# Exemple MCP Server filesystem (simplifié) : @server.call_tool("read_file") async def read_file(path: str): return open(path).read() # ← vrai code Python

04 — C'est quoi stdio ? Comment les deux process se parlent

stdio = standard input / output.

Quand un programme tourne, il a 3 "tuyaux" ouverts par défaut :

Tuyau Direction Exemple concret
stdin — standard input → vers le programme Ce que tu tapes au clavier dans un terminal
stdout — standard output ← depuis le programme Ce que le programme affiche à l'écran
stderr — standard error ← depuis le programme Les messages d'erreur (en rouge dans le terminal)

Comment l'agent et le serveur MCP se parlent via stdio

Analogie stdio

Imagine deux personnes dans deux pièces séparées, reliées par un tuyau pneumatique (comme à la banque).
L'agent glisse un message dans le tuyau (stdin du serveur).
Le serveur glisse la réponse en retour (stdout vers l'agent).
Pas de réseau, pas d'HTTP, pas de port — juste deux process qui s'échangent du texte directement sur la même machine.

Comment l'agent lance le serveur ?

L'agent spawn le serveur MCP comme un sous-process — il l'exécute lui-même au démarrage :

# L'agent fait l'équivalent de ça : subprocess.Popen(["python", "mon_serveur_mcp.py"]) # Puis il "parle" dans stdin/stdout de ce sous-process

Il existe aussi un mode HTTP/SSE quand le serveur tourne sur une autre machine (réseau), mais stdio est le transport par défaut en local.

07 — MCP est-il un protocole comme HTTP ?

Oui — l'analogie tient parfaitement. MCP est construit sur JSON-RPC 2.0, qui suit le même modèle requête/réponse qu'HTTP.

La différence : HTTP transporte n'importe quoi. MCP est spécialisé pour la communication agent ↔ outils — avec une discovery automatique pensée pour les LLMs.

HTTP MCP
Client envoie POST /search Agent envoie tools/call + nom de l'outil
Serveur reconnaît la route Serveur reconnaît le nom de l'outil
Serveur exécute le handler Serveur exécute la fonction Python/Node
Retourne 200 + JSON Retourne content: [{text: "..."}]
GET / pour voir les routes dispo tools/list pour voir les outils dispo
// Requête MCP (JSON-RPC 2.0) — même logique qu'un POST HTTP { "method": "tools/call", "params": { "name": "search_web", "arguments": { "query": "MCP protocol" } } } // Réponse MCP — même logique qu'un 200 OK { "content": [{ "type": "text", "text": "résultat..." }] }
En 3 phrases :
L'agent décide quoi faire — il raisonne et boucle jusqu'à atteindre l'objectif.
MCP est le protocole standard qui permet à l'agent de parler à n'importe quel outil sans coder une intégration sur mesure.
L'API est le service réel — MCP s'en sert en interne, l'agent n'a pas à la connaître directement.
