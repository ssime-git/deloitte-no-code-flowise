Architecture Decision Guide

Choisir le bon pattern architectural — du simple prompt jusqu'au fine-tuning.

01 — Arbre de décision

02 — Les 6 patterns

Prompt

🟤 Simple / Few-shot

Complexité: faible Latence: <2s

QUAND UTILISER

  • Tâche unique et bien définie
  • Classification, résumé, extraction
  • Structured output (JSON imposé)
  • Transformation de texte

NE PAS UTILISER SI

  • Données internes nécessaires
  • Plusieurs étapes interdépendantes
Frameworks : Instructor, Outlines, LiteLLM, OpenAI SDK direct
Workflow

🔵 Pipeline orchestré

Complexité: moyenne Latence: 5–30s

QUAND UTILISER

  • Étapes fixes et prévisibles
  • Process RH, legal, finance
  • Approbations humaines (HITL)
  • Audit trail requis

NE PAS UTILISER SI

  • Le chemin dépend du contenu
  • Besoin de décision dynamique
Frameworks : LangGraph, Prefect, n8n, Flowise, Temporal
RAG

🟢 Retrieval-Augmented

Complexité: moyenne Latence: 3–15s

QUAND UTILISER

  • Docs internes, KB, contrats
  • Données qui changent souvent
  • Besoin de sourcing / citations
  • Volume trop grand pour le contexte

NE PAS UTILISER SI

  • Données stables → fine-tune
  • Pas de corpus documentaire
Frameworks : LlamaIndex, LangChain, Weaviate, Qdrant, pgvector
Agent

🟣 Agent autonome

Complexité: haute Latence: 30s–5min

QUAND UTILISER

  • Objectif ouvert, chemin inconnu
  • Besoin d'itérer et corriger
  • Recherche, debug, analyse multi-sources
  • Tâches de niveau "dev junior"

NE PAS UTILISER SI

  • Latence critique (<3s)
  • Étapes connues → workflow
Frameworks : LangGraph, AutoGen, CrewAI, Hermes, Claude API
Agent + MCP

🟩 Agent + outils standardisés

Complexité: haute Latence: variable

QUAND UTILISER

  • Plusieurs agents partagent des outils
  • Outils maintenus par une autre équipe
  • Clients MCP-natifs (Cursor, Claude Desktop)
  • Découplage agent ↔ tooling requis

NE PAS UTILISER SI

  • 1 seul agent, 1 seul outil simple
  • POC ou prototype rapide
Frameworks : MCP SDK (Python/TS), Claude Desktop, Hermes, Cursor
Fine-tuning

🟠 Adapter le modèle

Complexité: très haute Inférence: rapide

QUAND UTILISER

  • Style / ton très spécifique (légal, médical)
  • Format de sortie propriétaire répétitif
  • Latence critique + domaine stable
  • Coût d'inférence à réduire (modèle plus petit)

NE PAS UTILISER SI

  • Données changent souvent → RAG
  • Pas de dataset d'entraînement (>500 ex.)
Frameworks : Unsloth, LoRA/QLoRA, OpenAI Fine-tuning API, Axolotl

04 — Composants clés

🧠

LLM

Moteur de raisonnement. GPT-4o, Claude, Gemini, Mistral, Llama.

🗄

Vector DB

Stocke les embeddings pour le RAG. Qdrant, Weaviate, pgvector, Pinecone.

🔢

Embeddings

Transforme le texte en vecteurs numériques. text-embedding-3, nomic, BGE.

🎭

Orchestrateur

Coordonne les étapes et les agents. LangGraph, CrewAI, AutoGen, Prefect.

💾

Memory

Contexte persistant entre sessions. Redis, Zep, Mem0, fichier JSON.

🔌

MCP Server

Expose des outils au LLM via protocole standard. Filesystem, GitHub, DB…

👤

HITL

Human-in-the-loop. Point d'approbation humaine avant action critique.

📊

Observability

Traces, logs, coûts. LangSmith, Langfuse, Arize, Helicone.

06 — Anti-patterns classiques

Anti-pattern

RAG quand il faut fine-tuner

Tu utilises RAG pour injecter du jargon métier statique que le modèle ne comprend jamais bien.

✓ Fix : fine-tune sur ce vocabulaire, garde RAG pour les données qui changent.

Anti-pattern

Agent quand il faut un workflow

Les étapes sont connues et fixes mais tu laisses l'agent "décider" — coûteux, lent, imprévisible.

✓ Fix : code un workflow explicite si le chemin est prévisible.

Anti-pattern

MCP pour un seul agent, un seul outil

Overhead inutile. Un tool() custom en 5 lignes suffit pour un POC ou un agent solo.

✓ Fix : MCP quand plusieurs agents partagent les outils ou en prod multi-équipe.

Anti-pattern

Fine-tuning comme premier réflexe

Fine-tuner sans essayer d'abord prompt engineering + RAG — 80% des cas se résolvent avant.

✓ Fix : prompt → RAG → fine-tune. Dans cet ordre.

Anti-pattern

Context stuffing sans RAG

Mettre 500 pages de docs dans le contexte système. Coûteux, dégradation de performance.

✓ Fix : RAG — ne récupère que les 3–5 chunks pertinents.

Anti-pattern

Agent sans observabilité

Agent en prod sans traces → impossible de débugger, coûts incontrôlables.

✓ Fix : LangSmith / Langfuse dès le premier jour en prod.

07 — Règles d'or

1

Commencer simple, complexifier par la preuve

Prompt → Workflow → RAG → Agent → Fine-tune. Chaque saut se justifie par une limite prouvée du niveau précédent.

2

Prévisible = workflow, incertain = agent

Si tu peux écrire les étapes sur un post-it à l'avance, c'est un workflow. Si le chemin dépend du contenu, c'est un agent.

3

RAG pour les données qui bougent, fine-tune pour le style qui reste

La connaissance changeante va dans un vector store. Le ton, le format, le vocabulaire propriétaire vont dans le modèle.

4

MCP = investissement d'équipe, pas de POC

MCP vaut le coût quand plusieurs agents ou équipes partagent les mêmes outils. En solo ou prototype : tool() custom en 5 lignes.

5

Observabilité avant scalabilité

Avant d'optimiser les coûts ou la latence, instrumente. Tu ne peux pas améliorer ce que tu ne mesures pas. LangSmith / Langfuse dès le jour 1 en prod.

Ordre de décision : Prompt → Workflow → RAG → Agent → Agent+MCP → Fine-tuning  ·  Chaque palier se justifie par une limite prouvée du précédent.
